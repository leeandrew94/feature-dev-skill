# Feature Dev

按 6 阶段将需求 → 代码 → 知识沉淀。

```
阶段0:需求进来 → 阶段1:读知识库 → 阶段2:需求澄清 → 阶段3:实现计划 → 阶段4:编码 → 阶段5:测试 → 阶段6:归档(手动)
```

**你是流程引擎**。项目的架构、规范、术语从知识库加载，按指令执行即可，不自行发挥。

---

## 阶段 0：需求进来

1. 优先 `拉取需求内容mcp服务名称` 按任务 ID 拉取需求；不可用则提示用户手动粘贴需求内容
2. 先暂存需求内容；任务文件在阶段 1 确认 `knowledge_root` 后写入知识库
3. `$knowledge_root/requirements/` 下已有该任务文件则读取合并，无则创建 `$knowledge_root/requirements/TASK_{id}_{描述}.md`
4. 文件含：任务 ID、标题、需求描述、涉及模块、验收标准、追踪表
5. 追踪表按"一条验收标准一行"控制粒度，不按函数或小改动拆分：
   - 字段：需求点、知识依据、改动位置、验证方式、归档位置
   - 阶段 0 先填需求点，其余字段填"待定"

---

## 阶段 1：读知识库

1. 定位 Obsidian 项目知识库：
   - `project_key=当前项目根目录名`（例如 `/Users/leeandrew/data/testCoding` → `testCoding`）
   - 读取本 skill 同目录的 `feature-dev.json`
   - `knowledge_root=feature-dev.json` 中的 `knowledge_base_root` + `project_key`
   - 只有 `knowledge_root/INDEX.md` 存在才视为有效知识库
   - 配置文件不存在、JSON 无效、`knowledge_base_root` 缺失或 `INDEX.md` 不存在时，询问用户输入知识库路径
2. 优先通过 `obsidian-mcp` 读取 `$knowledge_root/INDEX.md`
3. `obsidian-mcp` 不可用时，允许只读文件降级：
   - 默认读取 `$knowledge_root/INDEX.md` 和其索引到的 Markdown 文件
   - 如果配置读取不到有效 `knowledge_root`，或用户要求覆盖路径，则询问用户输入知识库路径
   - 用户输入路径后，仅在该路径存在 `INDEX.md` 时作为本次任务的 `knowledge_root`
4. 配置读取和用户确认后仍没有有效 `INDEX.md`，才跳过本阶段
5. 确认有效 `knowledge_root` 后，自动确保 `$knowledge_root/requirements/` 和 `$knowledge_root/tests/` 存在
6. INDEX.md 表格中「何时加载」列含"必读"/"all tasks"/"所有任务必读" → 无条件加载该行文件
7. 无「何时加载」列时，用（内容摘要 + 模块 + 标签）与需求描述做语义匹配 → 匹配到的加载
8. 支持 Obsidian 链接格式：`[[architecture.md]]`、`[[infrastructure/mysql.md]]`
9. 已加载文件内的 `[[双向链接]]` → 追读目标，目标路径相对 `knowledge_root`
10. 校验：知识库描述 vs 实际代码结构。不一致 → 标记为"疑似过期"，不得作为唯一实现依据
11. 输出"知识命中报告"：
   - 已加载：文件、命中原因、可用结论
   - 疑似过期：文件、冲突点、需要以代码为准的部分
   - 未命中但可能相关：文件或模块、为什么可能相关、是否需要用户确认

> **注意**：项目架构以知识库中的 `architecture.md` 为公共来源，编码规范以 `conventions.md` 为准。不读取当前项目内的 `wiki/`，也不依赖 `CLAUDE.md`、`AGENTS.md`、Cursor rules 等 agent 专属入口文件。知识库文件以 `INDEX.md` 命中项和已加载文件中的 `[[双向链接]]` 为加载边界。

---

## 阶段 2：需求澄清

1. 对照需求 + 已加载知识，列出不确定点
2. 向用户提问（优先选择题），用户回答后**立即**更新 `$knowledge_root/requirements/TASK_{id}.md`
3. 重复直到无疑问
4. INDEX.md 不存在时，增加以下澄清维度：项目分层和职责、涉及数据表/API、代码规范
5. 澄清导致验收标准变化时，同步更新追踪表的"需求点"

---

## 阶段 3：实现计划

1. 基于需求 + 知识库中的 `architecture.md` / `conventions.md`，输出改动计划
2. 用知识库中**实际的目录名和分层术语**，不套用通用名
3. 每个关键改动必须带知识依据：
   - 来自知识库 → 标注文件名，如 `conventions.md`、`infrastructure/mysql.md`
   - 来自代码现状 → 标注实际文件/类/函数
   - 来自用户澄清 → 标注"用户确认"
4. 包含：涉及文件清单、改动概要、数据变更（如有）、风险点、知识依据
5. 若某个关键决策没有知识依据，必须标记为"待确认假设"
6. 同步更新 `$knowledge_root/requirements/TASK_{id}.md` 追踪表：
   - "知识依据"填知识库文件、代码现状或用户确认
   - "改动位置"填计划涉及的文件/模块；尚未确定则填"待定"
7. **等待用户确认**，确认前不写代码

---

## 阶段 4：编码

1. 创建分支（命名从阶段 1 加载的规范来；无则用 `feature/TASK-{id}_{描述}`）
2. 严格按 `architecture.md` / `conventions.md` 声明的分层职责将代码放到正确目录
3. 遵循阶段 1 加载的项目规范（命名、错误处理、日志、接口返回格式等）
4. Commit 格式从加载的规范中来；规范未定义则不加限制

> **不内置任何分层规则、commit 类型、命名约定、签名行。一切由项目自行定义。**

---

## 阶段 5：测试

1. 生成 `$knowledge_root/tests/TASK_{id}_test.md`（用例、预期、实际结果）
2. 项目有测试命令则执行；失败则修复并更新测试文档
3. 同步更新 `$knowledge_root/requirements/TASK_{id}.md` 追踪表的"验证方式"
4. 输出测试报告摘要

---

## 阶段 6：归档

> 用户说"归档"或"沉淀知识"时触发，不自动执行。

1. 更新 `$knowledge_root/requirements/TASK_{id}.md` 补充实现逻辑和关键决策
2. 使用阶段 1 已确认的 `knowledge_root`
3. `INDEX.md` 不存在 → 只更新需求文件，跳过后续
4. 先输出"归档候选摘要"，等待用户确认后再写知识库：
   - 事实知识：表结构、接口、目录职责、配置项
   - 经验知识：本类需求的通用改法、风险、排查点
   - 决策知识：为什么选当前方案，不选其他方案
   - 历史需求：本次任务摘要、涉及模块、可复用点
5. 用户确认后，优先通过 `obsidian-mcp` 更新知识库
6. `obsidian-mcp` 不可用时，不自动写入 Obsidian 文件；只更新需求文件，并提示用户确认是否允许直接写入阶段 1 确认过的 `knowledge_root`
7. 按 INDEX.md 表格的行摘要/模块/标签定位目标文件来更新知识库：
   - 数据层变更 → 摘要含"表"/"schema"的行对应的文件
   - 架构变更 → 摘要含"架构"的行对应的文件
   - 新术语 → 摘要含"术语"的行对应的文件
   - 新决策 → `$knowledge_root/decisions/{序号}-{标题}.md`（ADR 格式）
8. INDEX.md「历史需求」表追加本次任务
9. 同步更新 `$knowledge_root/requirements/TASK_{id}.md` 追踪表的"归档位置"
10. 索引校验：INDEX.md 每行文件路径指向真实文件

---

## 降级 & 失败处理

| 场景 | 处理 |
|------|------|
| `拉取需求内容mcp服务名称` 不可用 | 提示用户手动输入需求 |
| `obsidian-mcp` 不可用 | 阶段 1 允许只读文件降级；默认路径无效时询问用户输入知识库路径；阶段 6 不自动写 Obsidian 文件 |
| `$knowledge_root/INDEX.md` 不存在 | 跳过阶段 1、6 的知识库读写；阶段 2 加重澄清 |
| 当前项目名和 Obsidian 项目目录名不一致 | 提醒用户当前使用目录名匹配规则，需要补充别名或配置 |
| INDEX.md 某文件缺失 | 静默跳过 |
| 阶段 4 失败 | git 回滚，知识库未改动 |
| 阶段 6 失败 | 提示用户确认，不回滚代码 |

---

## 质量门禁

1. 归档候选质量：
   - 事实知识必须能被代码、配置、接口或数据结构验证
   - 经验知识必须来自本次实现过程
   - 决策知识必须包含备选方案或取舍原因
2. 历史需求加载：
   - 默认只读 INDEX.md 中的历史需求摘要
   - 只有强相关时才加载完整历史需求文件
   - 最多加载 3 个历史需求文件
3. 追踪表闭环：
   - 阶段 5 结束时，每条需求点都必须有验证方式
   - 若无法自动验证，标记为"人工验证"并说明原因
